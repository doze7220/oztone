# PROJECT_CONSTITUTION.md
## AI駆動開発向け マスタードキュメント（プロジェクト憲法）

本ドキュメントは、AI-IDEが「OZtone」のコードを生成・修正する際の絶対的なルールである。AIはコード生成前に必ず本ドキュメントを参照すること。

### 1. プロジェクト前提知識
*   **プロジェクト名:** OZtone
*   **概要:** C++とWin32 APIで作成する、デスクトップアクセサリ風のMP3プレイヤー。
*   **特徴:** ウィンドウ枠を持たず、表示領域全面にアルバムアートとビジュアライザを描画するストイックな設計。

### 2. 絶対禁止事項（Win32 C++ アーキテクチャ維持ルール）
*   **モダンすぎるフレームワークの禁止:** 純粋なC++とWin32 API（または必要最小限のGDI/Media Foundation等）を使用し、巨大な外部UIフレームワーク（Qtなど）を勝手に導入してはならない。
*   **WndProcの肥大化禁止:** ウィンドウプロシージャ（WndProc）内に直接巨大なゲームロジックや描画処理（描画アルゴリズムそのもの）を書いてはならない。必ず専用のManagerやクラスへ処理を委譲すること。
*   **グローバル変数の乱用禁止:** Win32特有のグローバル変数の乱用を避け、極力クラスベース（ポインタ渡しやシングルトン等）で状態を管理すること。
*   **JSDoc/Doxygenスタイルの義務:** クラスやメソッドを新規作成する際は、必ずDoxygenスタイルのコメントを記述すること。

### 3. AI作業手順と REQ/RES 駆動開発ルール（絶対厳守事項）
本プロジェクトにおいて、新機能の追加やバグ修正を行う際、AIはコンテキストの混同や予期せぬコード書き換え（暴走）を防ぐため、必ず以下の **「REQ/RES 駆動開発プロセス」** に従って作業を実行すること。

#### 3.1. 命名規則と基本概念
* **REQ (Request)**: ユーザーからAIへのインプット（指示・要件定義・プロンプト）。
* **RES (Response)**: AIからユーザーへのアウトプット（実装計画書・作業レポート）。AIが生成する作業記録ファイルには、必ず `RES_` のプレフィックスを付けること。

#### 3.2. プロセス分離の原則（計画立案と実装実行の分割）
開発は必ず以下の2段階のステップを踏むこと。原則として、ステップごとにチャット（セッション）を新しくしてコンテキストをリセットする。

* **Step 1: 計画立案 (REQ -> RES)**
  * ユーザーからの REQ に基づき、高度なアーキテクチャ設計とタスクリストをまとめた「実装計画、兼作業レポート（RESファイル）」を新規作成する。
  * **【絶対厳守】このプロンプト内では、絶対にソースコードの修正（ファイルの書き換え）を行わないこと。計画立案のみに留めること。**
* **Step 2: 実装実行 (RES -> Code)**
  * ユーザーの承認後、出力された RES ファイルを引き継ぎ資料として読み込み、記載されたタスクリストに従ってコードの修正を実行する。
  * 作業は1回のレスポンスで1タスクと基本とする。プロンプトで「タスクX~Xまで実行すること」と明言されない限り、1タスクずつ処理すること。
  * 修正完了後、RESファイルのタスクリストのチェックボックスを完了 `[x]` に変更し、詳細作業内容をタスクごとに追記する。

#### 3.3. 実装計画、兼作業レポート (RES) 標準テンプレート
基本的に以下のフォーマットを使用し、AIはこれに完全に準拠してRESファイルを出力すること。

```text
# RES:実装計画・作業レポート Phase X-X: タスク名

## 1. 実装目的
(実装目的詳細)

## 2. アーキテクチャ設計
### 要件1:(要件名)
    - （内容詳細)

## 3. 実装タスクリスト
[ ] タスク1:(要件概略)
    - （作業内容詳細)

## 4. 詳細作業内容
### タスク1:(要件概略)
    - 詳細作業内容

## 5. HOTFIX1
### 原因・理由:(要件概略)
    - (要件詳細)

### 対応:(作業概要)
    - (作業詳細)
```
