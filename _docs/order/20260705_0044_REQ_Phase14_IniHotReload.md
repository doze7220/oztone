#### 【実装要件】
本タスクの目的は、アプリケーション実行中に `OZtone.ini` が外部のテキストエディタ等で上書き保存されたことを検知し、即座に設定を再読み込みして画面描画（レイアウトやフォント等）に反映させる「ホットリロード」を実装することである。

* **要件1: INIファイルのタイムスタンプ監視 (ConfigManager)**
  * `ConfigManager` に、`std::filesystem::file_time_type m_lastIniWriteTime;` のような最終更新日時を保持するメンバ変数を追加する。
  * `CheckForUpdates()` メソッドを新設し、`std::filesystem::last_write_time` を用いて更新を検知した場合に `true` を返し、内部のタイムスタンプを更新するロジックを実装する。
  * `LoadSettings()`（または同等の設定読み込み処理）を公開し、外部から再読み込みをトリガーできるようにする。
* **要件2: メインループでの定期監視 (Application)**
  * `Application::Run` のメインループ内において、一定間隔（例: 60フレームごと、あるいは1秒ごと）に `m_config->CheckForUpdates()` を呼び出す。
  * 更新が検知された場合、`m_config->LoadSettings()` を呼び出して最新の設定をメモリに反映させる。
* **要件3: キャッシュとリソースの動的再構築 (Renderer / Widgets)**
  * Phase 9-4, 9-5 のウィジェット化・リソースキャッシュ化の設計に従い、設定の再読み込み完了後は、UIの色やフォントサイズ、配置座標の変更を適用しなければならない。
  * `Application` から `Renderer` に対し、全リソースとレイアウトキャッシュの再生成を指示する処理（例: `Renderer::ReloadResources()` の新設、または各Widgetの `CreateResources` 呼び出しループなど）を実行する。
  * フォントサイズ変更等に対応するため、テキストレイアウトも確実に破棄・再生成されるように状態をクリアすること。

#### 【絶対遵守ルール (Constraints)】
* **低負荷の維持**: 毎フレームのファイルアクセスは避け、インターバルを設けることで負荷を抑えること。
* **既存アーキテクチャの尊重**: Widget 自身が直接 ConfigManager を監視するのではなく、Application が中枢として検知し、Renderer/Widget へ「再描画・再生成」の命令を下すトップダウンのデータフローを維持すること。
