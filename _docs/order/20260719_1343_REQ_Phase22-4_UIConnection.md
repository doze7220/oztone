#### 【実装要件】
完成したサムネイルエンジン（`ThumbCacher` および `ThumbnailDatabase`）をアプリケーション層およびUIの描画層と結線し、プレイリストの全曲サムネイル表示を稼働させる。

*   **要件1: サムネイル生成キューへのエンキュー (Application層)**
    *   `Application` クラスにて、プレイリストへ楽曲が追加されるタイミング（起動時のリスト復元時、およびドラッグ＆ドロップによる追加時など）で、追加されたファイルパスを `m_thumbCacher.Enqueue()` を用いてキューへ流し込む処理を実装する計画を立てること。
*   **要件2: プレイリストUIへのサムネイル描画 (Widget層)**
    *   `Widget_Playlist_DrawItems.cpp` 等におけるプレイリストの各アイテム（トラック）描画ループ内において、`m_thumbnailDatabase->DrawThumbnail` （または `GetThumbnailBitmap`）を呼び出し、データベースから取得したサムネイル画像を描画するフローへ変更する。
    *   サムネイル画像を描画するにあたり、既存のテキスト（曲名・アーティスト名など）のレイアウトが被らないよう、描画矩形（X座標のオフセット等）を自動調整するロジックを検討すること。

#### 【絶対遵守ルール (Constraints)】
*   **描画ループでの非同期維持**: `Widget` 層の描画ループ内でサムネイルを要求した際、データがキャッシュになくても（まだクックされていなくても）絶対にブロック（待機）してはならない。無い場合はフォールバック（ガラス板など）を即座に描画し、非同期で画像が準備されるのを待つこと。

#### 【指摘事項と解決方針】
*   **アーキテクチャ違反の指摘:**
    *   タスク4にて「WidgetContext にサムネイル取得用インターフェースを渡す」とあるが、これは Phase 21-9 等で確立した「Widgetの受動態化（データ層への直接アクセスや能動的なデータ取得の禁止）」に完全に違反している。
*   **正しい設計方針 (アンサー):**
    *   **タスク4の修正**: `Renderer_Context.cpp` にて、プレイリストの現在の可視範囲（スクロール位置等）から描画が必要なインデックス範囲を算出し、`ThumbnailDatabase::GetCachedThumbnailBitmap()` （非同期要求とキャッシュ取得）を呼び出す。取得できたビットマップのポインタを `std::unordered_map<size_t, ID2D1Bitmap*>` 等の形で `WidgetContext` へ詰め込み、データとしてWidgetへ渡す設計にすること。
    *   **タスク5の修正**: `Widget_Playlist_DrawItems.cpp` では、インターフェースを呼び出すのではなく、`WidgetContext` から渡されたマップ等を参照し、そこに画像が存在すれば描き、なければフォールバック（ガラス板）を描くだけの純粋な受動態の描画処理に徹すること。

#### 【ファイルの修正要件】
*   対象ファイル内の「タスク4」および「タスク5」の記述を、上記の方針に基づく設計へと断定的に書き換えること。
