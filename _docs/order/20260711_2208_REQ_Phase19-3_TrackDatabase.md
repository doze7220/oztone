###### 【実装要件】
現在、PlaylistManager が「プレイリストの曲順とフレーミング設定」および「楽曲のメタデータとFFTスキャン結果」の両方を管理し、1つのプレイリストファイル（.ozl）に保存している。これにより、非同期スレッド間のファイル書き込み競合（1曲目のFFT初期値上書きバグ等）が発生している。
本フェーズでは段階的なリファクタリングとして、「曲順とアルバムアートのオフセット設定」はプレイリスト側に残しつつ、「楽曲のメタデータとFFT解析結果」のみを完全に独立したデータベースへ分離する。

*   **要件1: TrackDatabase クラスの新設**
    *   楽曲固有の解析データ（曲名、アーティスト名、再生時間、`peakAmplitude`, `maxFrequency` 等）を管理する `TrackDatabase` クラスを新設する。
    *   キーをファイルパス（絶対パス）とし、値をメタデータとする連想配列でメモリ上に保持する。
    *   データの永続化先を `oztone_track.odb` (タブ区切りのTSV形式等) とし、実行ファイルと同階層等に保存・読み込みを行うメソッドを実装する。

*   **要件2: PlaylistManager の責務見直し（部分的分離）**
    *   `PlaylistManager` は、プレイリストの曲順（ファイルパス）と、現状の仕様である「アルバムアートのフレーミング情報（`artOffsetX`, `artOffsetY`, `artScale`）」の保持・管理を引き続き担当し、プレイリストファイル（.ozl）にはこれらの情報のみを保存するようスリム化する。
    *   メタデータの取得やFFT解析結果の更新ロジックは `TrackDatabase` 側へ分離・委譲する。

*   **要件3: スレッドセーフなデータ供給と解析フローの再構築**
    *   メインスレッド（再生やUI・ビジュアライザ描画）は、必要な時に `TrackDatabase` にファイルパスを渡して最新のメタデータ（FFT解析結果含む）を取得する構成とする（Read Onlyのアプローチ）。
    *   バックグラウンドの解析スレッド（`ParseThreadFunc` 等）は、追加されたパスが `TrackDatabase` に存在しない、または未解析（`isFFTLoaded == false` 等）の場合のみ波形フルスキャンを実行し、`TrackDatabase` のみを更新・ファイル保存する。
    *   これにより、即時再生時の「1曲目のFFT初期値上書きバグ（タイムパラドックス）」をアーキテクチャレベルで完全に根絶する。
