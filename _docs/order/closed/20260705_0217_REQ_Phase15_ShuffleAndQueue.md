#### 【実装要件】
本フェーズでは、ロゴ拡張メニューのシャッフルボタン（🔀）を機能させ、ストレート再生とシャッフル再生の切り替えを実装する。また、ファイル追加（D&D）時の挙動を整理し、ユーザーの意図に沿ったキュー管理を実現する。
これらを複雑な条件分岐で実装するのではなく、PlaylistManager に「キューの再構築」と「現在位置のワープ」という共通ロジックを持たせることで美しく解決する。

* **要件1: INI設定とシャッフルモードの追加**
  * `ConfigManager` に `ShuffleMode` (bool, デフォルトtrue) を追加し、保存・読み込みを実装する。
  * ロゴ拡張メニューのシャッフルボタン（🔀）のトグル状態をこの設定値と連動させる。
* **要件2: 究極のキュー再構築・ワープアーキテクチャ (PlaylistManager)**
  * `PlaylistManager` に `RebuildQueue(bool isShuffle)` メソッドを追加する。
    * 既存のキュー（および次周キュー）を完全に破棄し、`isShuffle` が true ならランダムに、false ならファイル登録順（0,1,2...の連番）で配列を作り直す仕様とする。以前のシャッフル状態は一切記憶しない。
  * `PlaylistManager` に `WarpToTrack(const std::wstring& filepath)` メソッドを追加する。
    * 再構築されたキューの中から指定されたファイルパスを探し出し、現在の再生インデックス (`m_shuffleIndex`) をそこに強制移動させる。
* **要件3: シャッフル切り替え時の挙動 (Application / Window)**
  * ロゴ拡張メニューのシャッフルボタンがクリックされた際、モードを反転してINIへ保存。
  * 即座に `RebuildQueue` を実行し、続けて「現在再生中の曲のパス」を用いて `WarpToTrack` を実行する。これにより、再生中の音楽を途切れさせることなく、プレイリストのUIだけがパキッと新しい順序に生まれ変わる。
* **要件4: D&D追加時の挙動改修 (Application)**
  * `OnFilesDropped` の処理内で `GetAsyncKeyState(VK_SHIFT)` を用いてSHIFTキーの押下状態を判定する。
  * **SHIFTキーが押されている場合:** 既存の仕様通り、ファイルを追加するだけでキューの再構築や再生曲の変更は行わない（BGM維持モード）。
  * **SHIFTキーが押されていない場合 (通常追加):** ファイル追加後、現在のシャッフルモードに従って `RebuildQueue` を実行し、「ドロップされた最初の曲」をターゲットとして `WarpToTrack` を行い、そこから即座に再生（`m_audioPlayer.Play`）を開始する。空の状態からの追加もこのロジックで統合される。

#### 【絶対遵守ルール (Constraints)】
* **既存アーキテクチャの尊重**: `LayoutCalculator` や `Widget` の描画ロジックは一切変更しないこと。裏側のインデックス配列（`m_shuffleIndices`）が書き換わるだけで、UIは自動的に追従する設計を維持する。
