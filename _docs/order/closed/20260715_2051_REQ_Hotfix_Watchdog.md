#### 【実装要件】
オーディオスレッドが原因不明のスピンループ等に陥った際、メインスレッドから再生位置の停滞を検知して自動でオーディオエンジンを蘇生させる「ウォッチドッグ機構」を実装する。

*   **要件1: ConfigManager へのパラメータ追加**
    *   `[System]` セクションに以下の設定を追加する。
        *   `WatchdogInterval` (デフォルト `1.0`): 通常時の再生位置ポーリング間隔（秒）。
        *   `WatchdogTimeout` (デフォルト `5.0`): 異常の疑いから「死亡」を確定するまでの沈黙時間（秒）。
    *   無限蘇生ループ（誤爆）を防ぐため、コード側で `WatchdogInterval` には下限 `0.5f`、`WatchdogTimeout` には下限 `2.0f` の `std::clamp` ガードを必ずかけること。

*   **要件2: ウォッチドッグ状態マシンの導入 (`Application`クラス)**
    *   内部状態として `Normal` と `Warning` を持つ列挙型（例: `WatchdogState`）を定義する。
    *   ポーリング用のタイマー変数、最後に確認した再生位置（`lastPosition`）、Warning状態に突入した時刻（`warningStartTime`）を保持するメンバ変数を追加する。

*   **要件3: 2段階監視ロジックの実装 (`Application::Run` または描画ループ周辺)**
    *   **【Normal 状態】**: `WatchdogInterval` 秒に1回だけ再生位置を取得する。オーディオが再生中であるにもかかわらず位置が変わっていなければ、`Warning` 状態へ移行し、`warningStartTime` を記録する。
    *   **【Warning 状態】**: 毎フレーム再生位置を監視する。値が少しでも進めば `Normal` へ戻る。値が進まないまま `WatchdogTimeout` 秒が経過した場合、「Dead」と確定し要件4のリカバリ処理を実行する。

*   **要件4: 既存の「デバイス復旧待ちモジュール」を利用した自己修復処理**
    *   Dead確定時、直ちに現在の再生曲のファイルパスと再生位置（秒数）を変数に退避する。
    *   `m_audioPlayer.Uninitialize()` でエンジンを強制破棄する。
    *   即座に再構築するのではなく、`Application.cpp` に既に実装されている **`m_isWaitingForDevice = true;` に設定し、メインループのデバイス復旧待機ロジックへ処理を合流させる。**
    *   これにより、`HasValidOutputDevice()` によって確実にデバイスが認識されてからエンジンが再構築・蘇生される安全なリカバリフローを実現する。

#### 【絶対遵守ルール (Constraints)】
*   **CPU負荷の最小化**: `Normal` 状態では毎フレームの再生位置取得を行わず、必ずインターバルごとのポーリングとすること。
*   **ダミーデバイスの回避**: リカバリ時のエンジン再構築は、必ずデバイスの存在確認が行われた後に行うこと。
