###### 【実装要件】
現在、PlaylistManager が「プレイリストの曲順・フレーミング設定」と「メタデータ・FFT解析結果」の両方を管理している。また、Application クラスがメインループの管理だけでなく、裏スレッド（ParseThreadFunc）を回してタグ解析やFFTスキャンを行うという重労働（神クラス化）を抱え込んでしまっている。
本フェーズでは、データ構造と解析ロジックをそれぞれ独立したクラスへ分離し、安全かつ堅牢なアーキテクチャへとリファクタリングする。

*   **要件1: TrackDatabase クラスの新設 (データ層の独立)**
    *   楽曲固有の解析データ（曲名、アーティスト名、再生時間、`peakAmplitude`, `maxFrequency` 等）を管理する `TrackDatabase` クラスを新設する。
    *   キーをファイルパス（絶対パス）とし、値をメタデータとする連想配列でメモリ上に保持し、`oztone_track.odb` (TSV形式) として保存・読み込みを行うメソッドを実装する。

*   **要件2: TrackAnalyzer クラスの新設 (解析スレッドの独立)**
    *   現在 `Application::ParseThreadFunc` にベタ書きされている「タグ解析」および「FFT事前スキャン」のロジック、およびキュー管理（`m_parseQueue`）、スレッド管理を `TrackAnalyzer` クラスとして完全に独立させる。
    *   `Application` は「曲が追加されたら `TrackAnalyzer` にパスを投げるだけ」というクリーンな状態に純化する。

*   **要件3: PlaylistManager と Application の責務見直し (部分的分離)**
    *   `PlaylistManager` は、プレイリストの曲順（ファイルパス）と、現状の仕様である「アルバムアートのフレーミング情報」のみを保持し、プレイリストファイル（.ozl）にはこれらの情報のみを保存するようスリム化する。
    *   メインスレッド（再生やビジュアライザ描画）は、必要な時に `TrackDatabase` からRead Onlyで最新のFFT解析結果を取得する構成とし、即時再生時の「1曲目のFFT初期値上書きバグ」をアーキテクチャレベルで根絶する。
