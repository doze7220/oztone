###### 【実装要件】
現在、個別のビジュアライザ（子）の中で生のFFTデータに対する対数マッピングやEQ処理を行っているため、曲ごとの最大周波数の違いによって描画が破綻するバグが発生している。これを解消するため、「親（Visualizerファサード）が 0 〜 maxFrequency の範囲を対数で256分割し、EQとクランプを済ませた完成済みデータを子へ渡す」というデータ指向のアーキテクチャへと改修する。

*   **対象ファイル: `src/Visualizer.cpp` (親ファサード)**
    *   `Visualizer::Draw` メソッドにおいて、子へ `spectrum` をそのまま渡すのではなく、以下の手順で256個の完成済みデータ (`processedSpectrum`) を生成すること。
    *   1. `std::vector<float> processedSpectrum(256, 0.0f);` を用意する。
    *   2. `i = 0` から `511` までのループを回し、0 〜 `maxFrequency` の間を対数スケールでサンプリングする。
        (例: `ratio = i / 511.0f`, `targetIndex = 対数補間によるインデックス計算`)
    *   3. 抽出した生の振幅値に対して、`ratio` (0.0〜1.0) に基づいて 5バンドEQ (`BandGain0` 〜 `100`) を滑らかに補間（Lerp）して乗算する。※Peak除算やピンクノイズは使用しない。
    *   4. 最後に `std::clamp(val, 0.0f, 1.0f)` を適用し、`processedSpectrum[i]` に格納する。
    *   5. 生成した `processedSpectrum` を、各スタイルの `Draw` メソッドへ渡す。

*   **対象ファイル: `src/Visualizer_PrismBeat.cpp`, `src/Visualizer_HaloDust.cpp` (子プラグイン)**
    *   各描画ロジックにおいて、渡された `spectrum` のサイズが常に「256」であり、すでに対数マッピング・EQ適用済みである前提に書き換えること。
    *   子側での複雑な `std::log10` などの対数計算や、`BandGain` の取得・適用処理をすべて削除し、単に256個のデータを画面幅（1200分割など）やバンド数（64等）に **線形マッピング** して描画するだけの純粋な描画ロジックへとリファクタリングすること。
