かなり良い。これならAI-IDEにとっても迷いが少ない指示になっている。

特に良い点は、

* **計画のみ・コード変更禁止**を最初に明記している
* **物理分割禁止**を最後にも書いている
* **「機能変更禁止」**を明示している

ので、AIが勝手に設計変更を始める可能性はかなり低い。

一方で、AI-IDEを長く使ってきた経験からすると、もう少しだけ追加するとさらに暴走しにくくなる。

---

## ① 「抽出のみ」を明記する

例えば

> DrawToolbar を新規実装する

ではなく

> **既存コードをそのまま抽出し、専用メソッドへ移動する**

という表現を入れる。

例えば

```
抽出したメソッドは既存コードを移動するのみとし、
ロジックの変更・簡略化・最適化は行わないこと。
```

これだけで、

AIが

> 「このif文まとめられますね！」

をやらなくなる。

---

## ② TrackVisualDataは"検討"を強くする

今の

```
TrackVisualData のような構造体を定義して
```

だと、

AIによっては

「じゃあ作ります」

になる。

ここは

```
TrackVisualData 等のデータ構造への移行可能性を検討し、
必要な設計案を計画書へ記載すること。
本フェーズでは導入は行わない。
```

くらいが安全。

---

## ③ Utility化も計画止まりを明示

同じ。

今だと

```
Utility化する。
```

に読める。

今回は計画書なので

```
Utility化候補を抽出し、
移行先および影響範囲を整理する。
本フェーズでは移行は行わない。
```

くらいが安心。

---

## ④ Drawの完成形を書いてしまう

AIはゴールが具体的なほど迷わない。

例えば

```
Draw() は最終的に概ね以下のような構成を目標とする。

Draw()
 ├ DrawGrip()
 ├ DrawBackground()
 ├ DrawToolbar()
 └ DrawPlaylistItems()
```

ここまで書いてしまっていい。

---

## ⑤ 将来の分割を明記

今回は整理フェーズだから、

```
本タスクは Phase20-14 以降で実施する物理ファイル分割の
前準備である。
```

と一文入れる。

これだけで

AIは

> 「ここでは整理だけ」

と理解しやすい。

---

## ⑥ 一番追加したい一文

これだけは入れたい。

```
抽出後の各メソッドは、可能な限り責務を単一化し、
将来「そのメソッド単位で .cpp ファイルへ移動できる」
粒度を目標とする。
```

これはAI-IDE向けにはかなり重要。

普通のリファクタリングなら

```
DrawToolbar()
```

だけでもいい。

でもAI向けなら

```
DrawToolbar()

DrawPinButton()

BuildToolbarText()
```

くらいまで責務を分けておく方が、
後から

```
PlaylistWidget_DrawToolbar.cpp
```

へコピーするだけで済む。

### 総評

この指示書は**90～95点**くらいの完成度です。

残りの5～10点は、「今回は**設計だけ・抽出だけ**」という境界をさらに明確にすることです。AIは「リファクタリング」という言葉から最適化や共通化まで踏み込みがちなので、「実装しない」「導入しない」「計画のみ」という境界を各要件にも繰り返し書いておくと、期待どおりの成果物になりやすくなります。
