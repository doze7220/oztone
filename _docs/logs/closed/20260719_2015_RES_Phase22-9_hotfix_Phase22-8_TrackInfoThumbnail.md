# RES:実装計画・作業レポート Phase 22-9: 3スロット式リングバッファの導入 (Phase 22-8 Hotfix)

## 1. 実装目的
Phase 22-8で改良を行ったドラムスクロール（サムネイルの非同期ロードと自己修復の軽量化）において、現在の「2スロット＋0.5でのフリップ方式」ではアニメーション開始直後のサムネイル表示が間に合わないなどの不備が発覚した。アーキテクチャの限界から2スロットでの解消が困難なため、これを廃止し「3スロット（リングバッファ）」を用いる。
画面外（完全に見えない位置）にあるスロットに対して、次に表示される曲のデータを「0ぴったり（整数位置到達時）」のタイミングで先行注入（Pre-fill）する完璧なリングバッファアーキテクチャを構築し、スクロール開始時からサムネイル非同期ロードが間に合う状態を実現する。

## 2. アーキテクチャ設計
### 要件1: DrumSlotの3スロット化
    - `src/WidgetContext.h` に定義されている `drumSlots` を `std::array<DrumSlot, 2>` から `std::array<DrumSlot, 3>` へ拡張し、リングバッファとして扱えるように再定義する。
    - 以前のPhase 22-8で追加されたサムネイルポーリング用の `thumbId` は維持し、3つのスロットすべてでサムネイル画像キャッシュと連携できるようにする。

### 要件2: TrackDrumの0ぴったり更新（リングバッファ化）
    - `src/Renderer_TrackDrum.h` および `src/Renderer_TrackDrum.cpp` の改修。
    - 従来の「0.5の境界でフリップとデータ取得を行う」処理を完全に廃止。
    - アニメーション開始時、およびスクロールが整数位置（スロットが中央に到達した瞬間 = 0ぴったり）に到達した際に、画面外に押し出された（完全に見えなくなった）スロットのインデックスを再利用する。
    - `dataProvider` から「次に画面に入ってくるスロット」のデータを取得し、画面外のスロットに対して先行注入（Pre-fill）するリングバッファロジックを実装する。
    - 【絶対遵守】エンキューの絶対禁止（発注は司令塔のみ）。0ぴったり更新のタイミングで `dataProvider` を呼ぶ際、`isNew == true` であってもサムネイル生成の新規発注（`EnqueueTrack`等）を絶対に行ってはならない。サムネイル工場の暴走を防ぐためである。

### 要件3: TrackInfoにおける3スロットの描画と物理マージンの禁止
    - `src/Widget_TrackInfo.cpp` の改修。
    - 描画ループを改修し、コンテキストから渡される3つの `DrumSlot` 全てを描画対象とする。
    - 各スロットの現在の相対座標から計算し、見えているものだけがD2Dのクリッピングで自動的に描画される純粋なループへと純化する。
    - 【絶対遵守】余白や見えない枠のための特殊な座標計算（物理マージン）は実装せず、純粋な3つのスロットのリングバッファの循環と、相対距離からの座標計算のみで完結させること。
    - 【絶対遵守】Widgetは「究極の受動態」であるため、自身で画像取得（`GetCachedThumbnailBitmap`等のポーリング）を絶対に行わず、コンテキストの `artBitmap` をそのまま描画することに徹すること。

### 【HOTFIX 追記】Phase 22-9 HOTFIX 1: ドラムスクロール方向の反転
*   **事象**: 「次の曲」へ進む際（ドラムのインデックスが進行する際）に「上から下へ落ちてくる」動きになっていた。これをプレイリストの進行に合わせた「下から上へせり上がる」動きにする必要があった。
*   **対応**: `src/Widget_TrackInfo.cpp` の `Draw` メソッド内の3スロット描画ループにおいて、Y座標オフセットの計算を `double diffOffset = ctx.drumRelativePosition - static_cast<double>(i);` と符号を反転させることで、直感的な動きに修正した。
*   **制約遵守状況**: TrackDrumエンジン側（演算層）には一切触れず、描画層である `Widget_TrackInfo.cpp` の座標マッピングのみで問題を解決した。

### 【HOTFIX 追記】Phase 22-9 HOTFIX 2: リングバッファ初期化とベクトル進行の完全修復
#### 原因・理由: 描画更新時のインデックス計算バグと開始時のPre-fill不備
    - `StartDrumAnimation` 実行時、開始位置ではなく目標インデックスだけをプレフィルしていたため、ジャンプ間のアニメーションで正しく連番が表示されなかった。
    - アニメーション更新（`Update`）時における進行方向と絶対インデックスの計算を、「現在の絶対位置(中央) + 進行方向」としてより明確に整理する必要があった。

#### 対象ファイル: `src/Renderer_TrackDrum.cpp`

#### 対応: 初期化ロジックの修正と更新ロジックの整理
    - `StartDrumAnimation` 内の初期化処理において、目標インデックスではなく「アニメーション開始前の現在位置（`startPos`）」を基準に、進行方向に向かって3スロット分を正しく連番で事前注入（Pre-fill）するように修正した。
    - `Update` 内の更新処理において、0ぴったりでの更新時に「現在の絶対位置(中央) + 進行方向」という正しい数式を用いてPre-fillが行われるようロジックを整理・明確化し、物理法則に完全に準拠した動作を保証した。
    - 描画層への影響を排除し、エンジン側で絶対的なカプセル化を維持しつつ、エンキューを一切伴わないコールバックによるデータ更新を徹底した。

### 【HOTFIX 追記】Phase 22-9 HOTFIX 2-B: リングバッファ計算式の完全パージと純化
#### 原因・理由: 逆算バグによる逆再生（5→4→3）の発生
    - TrackDrumエンジンにおいて、Pre-fillするインデックスを決定する際、`targetIndex`（目標）や `distance`（残り距離）を用いた逆算が行われており、これが原因で1→3へのジャンプ時にサムネイルや曲名が「5→4→3」と逆再生される致命的バグが発生していた。
    - アニメーションの進行中におけるプレフィルの算出が、現在の絶対位置を基準としておらず、計算式に無用な複雑さ（逆算）が混入していたため。

#### 対象ファイル: `src/Renderer_TrackDrum.cpp`

#### 対応: 逆算ロジックの完全パージと現在地ベースの連番注入
    - `Update` 関数において、進行方向（`direction`）の決定を `m_drumTargetPosition` を使わず、純粋に「現在の絶対位置(中央)と前回の整数位置との差分 (`currentPosRound - m_lastIntegerPosition`)」で算出するよう修正した。
    - `PrefillAbsolute` 関数内から、目標位置を用いた逆算（`absolutePos - m_drumTargetPosition`）を完全にパージした。
    - 代わりに、引数として渡される `absolutePos` そのものをそのまま `dataProvider` に渡すことで、「現在の絶対インデックスの整数値 ＋ 必要なスロットのオフセット」だけのシンプルな計算値がそのまま注入される純粋な足し算へと純化させた。
    - 描画層（`Widget_TrackInfo.cpp`）への改修は一切行わず、サムネイルID取得時のエンキューも厳格に禁止（パージ）したまま対応を完了した。

### 【HOTFIX 追記】Phase 22-9 HOTFIX 2-C: 逆ベクトルに対する相対インデックスの論理反転（物理座標とプレイリスト座標の同期）
#### 原因・理由: 常に特定のインデックスへ回帰してしまう不具合の発生
    - HOTFIX 2-B において「逆算のパージ」という方針を極限まで適用し、物理座標（`absolutePos`）をそのまま注入した結果、物理座標とプレイリスト座標（`currentIdx`）が完全に乖離してしまった。
    - 「次の曲（NEXT）」へ進む際、UIのスクロール方向としてドラムの物理座標（`m_drumTargetPosition`）は減少し、プレイリストのインデックス（`currentIdx`）は増加する、という**逆ベクトル**の関係にあった。
    - この逆ベクトル関係を無視して `absolutePos` を直接渡したため、数式 `targetIdx = currentIdx + absolutePos` が常に一定の定数（例：28曲目）に回帰してしまい、曲移動時に一瞬だけ本来の曲に切り替わった直後、また同じ曲へとスクロールしてしまう現象が発生した。

#### 対象ファイル: `src/Renderer_TrackDrum.cpp`

#### 対応: 逆ベクトルを考慮した純粋な差分計算の再適用
    - `PrefillAbsolute` 関数において、物理座標系から論理インデックス系への正しいマッピングを行うため、`int relativeIndex = m_drumTargetPosition - absolutePos;` を適用した。
    - 以前の逆算バグ（5→4→3）は `absolutePos - m_drumTargetPosition` と符号が逆であったことが原因であり、`m_drumTargetPosition` の存在自体が悪だったわけではないと判明した。
    - UI側が要求するスクロール方向（座標減少）とプレイリスト進行（インデックス増加）の逆ベクトルを完全に相殺するこの `m_drumTargetPosition - absolutePos` という数式により、目標曲への到達と連番インデックスの完璧な同期を達成した。
    - なお、`Update` 関数内の `direction` 計算における `targetIndex` 排除（`currentPosRound - m_lastIntegerPosition`）は正しく機能しており、これはそのまま維持した。
