# RES:HOTFIX作業レポート: ClampArtFraming の完全再実装と実装漏れの原因調査

## 1. 調査目的
`Renderer::ClampArtFraming` のクランプ処理がすでに働いていた件、および「右クリック＋ホイール」による画像拡大縮小機能の実装漏れが発生している原因について、Phase 18-12  당시の実装ログ（`20260707_2345_RES_Phase18-12_ArtFraming.md`）とBackgroundManager導入によるアーキテクチャ変化を踏まえて調査し、原因をまとめる。

## 2. 調査内容と実装漏れの原因

過去の実装履歴と現在のコードベースを比較した結果、「右クリック＋ホイール」による拡大縮小機能が正しく動作していない（実装漏れと見なされる）原因は、**「BackgroundManager への移行と大解体による、ズーム連動計算とクランプ連携の喪失」** にあります。具体的には以下の3点が複合しています。

### 原因1: ズーム時のオフセット補正（ズーム中心の維持）の欠落
Phase 18-12 当時は、`LayoutCalculator::CalculateBackgroundLayout` 内で描画先矩形（`destRect`）を画面外へはみ出させる形で計算し、`CalculateArtFramingBounds` によってクランプ範囲を動的に決定していました。
しかし、今回の機能復活時に設定された `Application_Initialize.cpp` のコールバック（`SetArtFramingScrollCallback`）では、単に `artScale` に固定値を加減算しているだけで、**「スケール変更に伴い、マウスポインタ（または画面中央）の位置を維持するように `artX`, `artY` を補正する」という拡大縮小の本質的な計算ロジックが欠落**しています。
これにより、ホイールを回しても左上や中央など固定の原点を基準に拡縮されてしまい、画像ビューアとして期待される直感的なズーム挙動になっていません。

### 原因2: `ClampArtFraming` 内での `scale` 参照更新の漏れ
現在 `Renderer::ClampArtFraming(float& scale, float& offsetX, float& offsetY)` として再実装されたメソッドでは、`offsetX` と `offsetY` についてはクランプ後に引数を書き換えて呼び出し元（`Application_Initialize.cpp` 側のローカル変数、ひいてはDBへの保存値）にフィードバックしています。
しかし、**`scale` については内部の計算に利用するだけで、クランプ（最小値ガードなど）を適用して呼び出し元へ書き戻す処理が一切書かれていません。**
その結果、スケールの限界制御が Renderer 側で行われず、Application 側と分離された状態になっています。

### 原因3: BackgroundManager 導入による状態の乖離
Phase 18-12 当時は BackgroundManager が存在せず、Renderer が直接画像を管理し、描画矩形の計算とフレーミングのクランプが密結合していました。
フェーズ23で画像管理が BackgroundManager に分離され、一時的にフレーミング機能がパージされた後、今回の Hotfix で「コールバックの再結線」と「`ClampArtFraming` の復活」という形で表面上のロジックだけが繋ぎ直されました。
しかし、当時の `LayoutCalculator` 内部で行われていた「スケールとオフセットの密接な連動計算」が十分に復元されなかったため、拡大縮小機能が不完全な状態（実装漏れ）として残ってしまいました。

## 3. 今後の修正方針（今回は調査のみ）
上記の原因から、「右クリック＋ホイールでの画像拡大縮小」を正しく実装するためには、以下の修正が必要になると考えられます。
1. `Window_Mouse.cpp` からコールバックへ、マウスの X, Y 座標を渡すようにシグネチャを変更する。
2. コールバック内で、マウス座標をズーム中心として `artX`, `artY` を補正するスケール計算ロジックを実装する。
3. `Renderer::ClampArtFraming` 内に、`scale` のクランプ（`scale = std::max(1.0f, scale);` など）を追加し、参照変数を正しく書き戻す。

（※本レポートは原因調査に基づく報告であり、実際のコード修正は含まれていません。）
